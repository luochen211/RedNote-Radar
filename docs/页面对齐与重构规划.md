# 页面对齐与重构规划

更新日期：2026-03-17  
主参考文档：`网页开发需求整合V7.docx`  
辅助参考：`docs/网页开发需求V2.2.md`、`docs/网页开发需求整合.md`、`extracted_web/网页代码/*.html`

## 1. 规划目标

本规划用于统一当前项目的后续改版方向，解决以下问题：

- 以 `网页开发需求整合V7.docx` 作为当前项目实现的主基线。
- 以解压得到的 HTML 原型作为视觉与信息层级参考，而不是机械照搬。
- 明确哪些页面优先做“视觉对齐”，哪些页面优先做“信息结构对齐”，哪些页面保持当前实现即可。
- 避免继续出现“文档口径、原型口径、现有代码口径”三套并存的情况。

最终目标是让当前 Next.js 项目同时满足三件事：

- 页面结构与功能符合 V7 文档。
- 重要页面的视觉完成度接近原型稿。
- 当前已有的真实路由、鉴权、数据流不被原型稿的单页 Demo 结构拖回去。

## 2. 参考优先级

后续改版统一按以下优先级执行：

1. `网页开发需求整合V7.docx`
2. 当前项目真实代码结构与已有接口
3. `extracted_web/网页代码/*.html` 原型稿
4. `docs/网页开发需求V2.2.md`
5. `docs/网页开发需求整合.md`

解释：

- V7 已经明确写明“按当前项目实现更新”，因此它不是纯需求草稿，而是更接近验收基线。
- HTML 原型的价值在于视觉风格、卡片布局、信息密度、组件层次，不在于页面组织方式。原型是单页切换，当前系统应继续保持多路由。
- V2.2 和整合版适合作为补充材料，尤其用于方法介绍、指标解释、部署建议和部分措辞。

## 3. 当前现状判断

### 3.1 已经做对的部分

- 项目已经完成从“单页原型”到“多路由系统”的拆分。
- 登录、上传、预测、分析、历史、管理后台都已经有真实路由。
- 后台已有用户、提交、审计与统计接口，方向与 V7 一致。
- 上传页字段已接近 V7，且已支持默认封面兜底。
- 分析页已经具备四个大类：Quality、Sentiment、Consistency、Oriental/Eastern Aesthetics。

### 3.2 目前最明显的问题

- 首页、Overview、History 三页的视觉风格不统一。
- 文案一度混入临时产品化描述，没有完全按 V7 和原始方法介绍来写。
- Overview 页是白底文档页，但曾被全局深色主题串色，说明样式边界还不够稳。
- 底部友情链接区在不同页面下表现不一致，说明组件与页面主题的职责尚未完全分开。
- History 页与原型相比完成度偏低，信息层级、封面展示、筛选与操作感不足。
- 分析页虽有功能，但“报告页”的信息组织和视觉节奏仍弱于原型。

## 4. 页面对齐策略

### 4.1 首页 `/`

目标定位：

- 按 V7 作为“系统入口与身份认证页”。
- 左侧保留方法名称、简介和核心能力摘要。
- 右侧为登录/注册卡片。
- 视觉上参考 `登录页面.html`，但内容口径以 V7 为准。

当前问题：

- 首页已经加入酒店背景图和玻璃面板，但文案、排版和信息密度仍在调试中。
- 部分内容更像“展示页”，而非“登录入口页”。

改版原则：

- 强化“登录优先”，弱化不必要的长篇说明。
- 保留方法名称、方法简介、五大模块摘要即可，不在首页塞过多正文。
- 首页必须一屏看完，不依赖纵向滚动。
- 左侧视觉应更像“品牌封面”，右侧更像“可操作工作台入口”。

建议动作：

- 压缩左侧说明文案为 1 段方法简介 + 4 张能力卡。
- 统一首页英文/中文标题排版，保证白字在背景图上足够清晰。
- 登录卡中“用户登录 / 管理员登录 / 申请注册”层级再清晰化。
- 首页底部不再承担大量友情链接内容，避免首屏被拖长。

### 4.2 系统介绍页 `/overview`

目标定位：

- 按 V7 作为“系统能力与方法论展示页”。
- 这是最接近文档展示页的页面，应允许白底黑字。

当前问题：

- 与首页的深色科技风切换较大，但这种切换本身不是问题，关键是风格需要稳定。
- 当前 Methodology、Modules、FooterLinks 的信息方向基本正确，但文案和层级还可以进一步向 V7 对齐。

改版原则：

- Overview 页采用“正式文档展示风格”，以可读性优先。
- 架构图、五模块说明、合作与友情链接三段必须清晰分区。
- 视觉上可保留高级感，但不追求首页那种重氛围。

建议动作：

- 顶部说明区按 V7 重写成“方法名 + 一句话简介 + CTA”。
- 方法介绍区重写为五大模块卡片，不再混入临时的工程说明文案。
- 架构图区保留占位，但必须明确未来替换真实图的方式。
- 底部友情链接区保持黑底，与 V2.2 中“Footer 黑色底栏”一致。

### 4.3 数据上传页 `/upload`

目标定位：

- 按 V7 作为“核心数据录入页面”。
- 保持当前真实表单与提交流程。

当前问题：

- 功能接近完成，但视觉精致度与原型有差距。
- 原型的上传区域更像产品化控制台，当前页面偏实用。

改版原则：

- 不动当前真实字段与提交逻辑。
- 只强化布局、字段说明、拖拽上传反馈和封面预览展示。

建议动作：

- 参考 `酒店短视频智算平台-Upload页重构.html` 调整上传卡片层次。
- 重点突出 9 类输入的分组关系。
- 将 `Account follower count / following count / total likes and bookmarks` 这类英文字段严格对齐到 V7 与会议修订意见。
- 提交按钮区加入更明确的处理中状态反馈。

### 4.4 预测页 `/prediction/[id]`

目标定位：

- 按 V7 作为“任务处理中 + 双范围预测结果页”。

当前问题：

- 功能方向是对的，但展示还偏基础。
- 与原型“仪表盘报告感”相比，页面张力不足。

改版原则：

- 保留现有轮询逻辑。
- 将 UI 提升为更明确的“两块预测大盘 + 文本说明 + 当前任务摘要”。

建议动作：

- 参考 `登录页面.html` 中 dashboard 区和 `分析报告详情页.html` 的预测区。
- 强化两个仪表盘卡片的一致性与视觉对比。
- 对当前上传内容摘要做更稳定的排版，支持封面缩略图。

### 4.5 分析页 `/analysis/[id]`

目标定位：

- 按 V7 作为“多维分析报告页”。
- 这是系统最核心的结果页。

当前问题：

- 已有四个 Tab 和 tooltip，但总体“报告感”“诊断感”仍弱。
- 与原型的层次差距主要体现在：标题区域、摘要区、模块节奏、卡片精细度。

改版原则：

- 不先追求花哨动效，先把报告结构做完整。
- 顶部应固定展示 Local / Global 两个预测结果。
- 四个分析区内部统一视觉语言，但每类图表应有差异化。

建议动作：

- 参考 `酒店短视频智算平台-分析报告详情页.html`。
- Quality：环形图 + Yes/No 指示卡。
- Sentiment：评分卡或简洁柱形结构。
- Consistency：条形分段卡片。
- Eastern Aesthetics：双三角雷达图继续保留，但标题统一使用 `Eastern Aesthetics`。
- 所有指标名称的 tooltip 文案，以整合文档中的变量解释为准。

### 4.6 历史记录页 `/history`

目标定位：

- 按 V7 作为“用户历史任务入口”。

当前问题：

- 当前页面功能有，但展示明显简化。
- 与原型相比，缺少封面缩略图、过滤感、操作密度和卡片感。

这是当前最值得优先补强的一页。

建议动作：

- 参考 `酒店短视频智算平台-历史记录页交互版.html` 重构。
- 列表项加入封面图、标题、时间、状态、预测摘要、跳转按钮。
- 支持至少一个基础筛选维度：状态。
- 已完成任务跳 `analysis`，处理中任务跳 `prediction`，保持现有逻辑。

### 4.7 管理后台 `/admin`

目标定位：

- 按 V7 作为运营和审计中台。

当前问题：

- 功能方向正确，但表现方式仍偏“基础表格页”。

改版原则：

- 先不做过多视觉实验，以管理效率和可读性优先。
- 保持三大 Tab：用户、提交、审计统计。

建议动作：

- 优化 KPI 卡、筛选栏、表格头部和导出入口布局。
- 后续如果时间有限，后台不作为第一优先级视觉重构页面。

## 5. 优先级排序

### P0：立即对齐

- 首页 `/`
- 系统介绍页 `/overview`
- 历史记录页 `/history`
- 分析页 `/analysis/[id]`

原因：

- 这四页最直接影响“老师/客户/甲方第一眼观感”。
- 当前用户已经对首页、Overview、友情链接、文案多次提出修改。

### P1：紧随其后

- 上传页 `/upload`
- 预测页 `/prediction/[id]`

原因：

- 功能已经基本成型，主要是视觉和信息组织的打磨。

### P2：稳定期再做

- 管理后台 `/admin`
- Profile 页及其它补充页

原因：

- 对外展示优先级低于首页、介绍页、分析页和历史页。

## 6. 执行顺序建议

建议按以下顺序实施：

1. 锁定全局设计规则  
   包括导航、页面宽度、卡片圆角、按钮、白底文档页、黑底链接区、深色工作台页三套基础主题。

2. 完成首页改版  
   解决首屏、一屏展示、左图右卡、白字可读性、文案口径问题。

3. 完成 Overview 改版  
   将其稳定为“方法论文档页”，彻底解决串色和内容杂糅。

4. 重构 History 页  
   用原型稿方式补齐卡片、封面、状态、操作。

5. 提升 Analysis 页  
   重点做报告结构和图表区节奏。

6. 微调 Upload / Prediction  
   让上传和预测页与上述页面风格闭环。

7. 最后整理 Admin  
   做中后台层级优化。

## 7. 统一设计原则

后续所有页面必须遵守以下约束：

- 首页是“品牌化入口页”，不是长文档页。
- Overview 是“正式介绍页”，不是首页的复制。
- 上传、预测、分析、历史属于同一套“工作台风格”。
- 友情链接区在所有出现位置都使用统一黑底版本。
- 文案优先遵循 V7，不再临时凭感觉改。
- 中英文命名保持一致，不混用 `Oriental` / `Eastern`。

## 8. 命名与口径统一

当前建议统一如下：

- `Oriental Aesthetics` 统一改为 `Eastern Aesthetics`
- `M_Oriental_Culture` 页面展示名为 `Eastern`
- `项目顾问、合作酒店与联系方式` 统一为 `友情链接`
- 各类外部分发渠道统一归为 `社交媒体平台`
- 首页与 Overview 页的方法名称统一为：
  `Video-based Social Media Marketing Analysis and Effectiveness Prediction`

## 9. 验收标准

完成本轮改版后，应满足以下标准：

- 首屏首页不需要纵向滚动即可看完主要内容。
- 首页、Overview、History、Analysis 四页风格统一且边界清楚。
- Overview 页正文全部可读，不再出现白底浅字问题。
- 友情链接区在所有页面都是黑底版本。
- History 页至少达到原型 70% 的信息密度。
- Analysis 页的四个 Tab 与 V7 文档一致。
- 页面中的重要标题、字段名、模块名与 V7 口径一致。

## 10. 建议的下一步工作

如果按本规划继续推进，下一阶段建议直接开始以下任务：

1. 完成首页定稿  
2. 重构 Overview 页文案与方法模块  
3. 重构 History 页卡片列表  
4. 统一 Analysis 页标题、报告头与图表风格
